System Requirements Specification for Maintenance Web Tool (MaT)

1. Introduction

1.1 Purpose

The purpose of this document is to specify the software requirements for the Maintenance Web Tool (MaT), a full-stack web application for managing maintenance and repair services for motorcycles, bicycles, and scooters. The application provides functionality for both clients and administrators, including user authentication, appointment handling, supply and order management.

1.2 Scope

MaT is a web-based platform built with HTML, CSS, JavaScript (frontend), Node.js (backend), and PostgreSQL (database). It enables clients to make appointments and track their service history, while administrators can manage supplies, approve requests, and handle orders. It is structured following the REST architectural style.

1.3 Definitions, Acronyms, and Abbreviations

  • JWT: JSON Web Token
  • REST: Representational State Transfer
  • API: Application Programming Interface
  • UI: User Interface

2. Overall Description

2.1 Product Perspective

MaT is a standalone product consisting of a front-end client communicating with a back-end server over HTTP. It is modular in design, enabling role-based access control for different types of users.

2.2 Product Functions

  • User registration and authentication via JWT
  • Client-side dashboard for managing appointments and service records
  • Admin-side interface for reviewing appointments and managing inventory
  • RESTful API endpoints for CRUD operations on appointments, supplies, and orders

2.3 User Classes and Characteristics

  • Client: Can register, login, make appointments, and view personal account data
  • Admin: Has all client capabilities plus the ability to manage supplies, orders, and user roles

2.4 Operating Environment

  • Front-end: Modern web browsers (Chrome, Firefox, Edge)
  • Back-end: Node.js runtime on a server, PostgreSQL as the database engine

2.5 Design and Implementation Constraints

  • Backend uses only the built-in http module in Node.js (no Express)
  • Data must be persisted in a PostgreSQL database
  • JSON must be used for API communication

2.6 Assumptions and Dependencies

  • Node.js and PostgreSQL are installed and correctly configured
  • Environment variables are used for configuration (e.g., JWT_SECRET)

3. Specific Requirements

3.1 Functional Requirements

  • FR1: Users shall be able to register and login securely
  • FR2: Clients shall be able to create and view their appointments
  • FR3: Admins shall be able to approve, reject, or review appointments
  • FR4: Admins shall be able to manage supplies and create orders
  • FR5: JWT shall be used for securing routes

3.2 Non-Functional Requirements

  • NFR1: The system shall respond to client requests within 2 seconds
  • NFR2: The frontend shall be mobile responsive
  • NFR3: The backend shall validate all user inputs and return appropriate status codes
